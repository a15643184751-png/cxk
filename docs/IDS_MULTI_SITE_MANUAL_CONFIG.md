# IDS 双机两站点手工配置说明

## 1. 适用场景

这套方案适用于下面这种真实部署方式：

- IDS 单独部署在一台机器上，例如 `10.134.32.142`
- 业务网站部署在另一台机器上，例如 `10.134.32.132`
- 网站机器上同时有两个网站
- 客户端统一访问 IDS 机器，不直接访问网站机器
- 不依赖域名、`hosts`、内网 DNS，只使用 `IDS_IP + 入口端口`

## 2. 拓扑示例

示例角色如下：

- IDS 机器：`10.134.32.142`
- 网站机器：`10.134.32.132`

IDS 机器上运行：

- IDS 前端：`5175`
- IDS 后端：`8170`
- 网站 1 入口：`8188`
- 网站 2 入口：`8288`

网站机器上运行：

- 网站 1 前端：`5173`
- 网站 1 后端：`8166`
- 网站 2 前端：`5273`
- 网站 2 后端：`8266`

访问路径如下：

- 客户端访问 `http://10.134.32.142:8188`
  - 先进入 IDS 网关
  - 再转发到 `http://10.134.32.132:5173` 和 `http://10.134.32.132:8166`

- 客户端访问 `http://10.134.32.142:8288`
  - 先进入 IDS 网关
  - 再转发到 `http://10.134.32.132:5273` 和 `http://10.134.32.132:8266`

如果第二个网站是单体服务，不区分前后端，那么把它的 `frontend_base_url` 和 `backend_base_url` 都写成同一个地址即可，例如：

```env
http://10.134.32.132:8080
```

## 3. 为什么这里用不同入口端口

在不使用域名和 `hosts` 的前提下，同一个 IP 的同一个端口无法区分两个不同网站。

所以这里采用的是最稳妥、最容易落地的做法：

- `IDS_IP:8188` 对应网站 1
- `IDS_IP:8288` 对应网站 2

这样后续换了 IP 也没关系，只要把 IDS 机器当前 IP 换进去即可。

## 4. 运行配置脚本

在 IDS 机器上执行：

```bat
D:\ids\runtime\startup\configure-ids-multisite.bat
```

或者执行：

```powershell
D:\ids\runtime\venvs\ids-backend\Scripts\python.exe D:\ids\runtime\startup\configure_ids_multisite.py
```

## 5. 按示例拓扑填写

如果你的环境就是：

- IDS：`10.134.32.142`
- 网站机：`10.134.32.132`
- 网站 1：`5173 / 8166`
- 网站 2：`5273 / 8266`

那么输入建议如下：

```text
配置文件路径: D:\ids\ids-standalone\ids-backend\.env
IDS 机器当前 IP: 10.134.32.142
默认站点标识: campus_supply_chain
默认站点入口端口: 8188
默认站点前端上游地址: http://10.134.32.132:5173
默认站点后端上游地址: http://10.134.32.132:8166
除默认站点外，还要接入几个网站: 1

--- 额外入口 1 ---
站点标识（site_key）: site2
站点显示名: site2
入口端口: 8288
该站点前端上游地址: http://10.134.32.132:5273
该站点后端上游地址: http://10.134.32.132:8266
```

## 6. 写入后的 `.env` 应该是什么样

上面的输入会生成类似下面的配置：

```env
IDS_GATEWAY_DEFAULT_PORT=8188
IDS_GATEWAY_FRONTEND_BASE_URL=http://10.134.32.132:5173
IDS_GATEWAY_BACKEND_BASE_URL=http://10.134.32.132:8166
IDS_GATEWAY_PORT_MAP=[{"site_key":"site2","display_name":"site2","ingress_port":8288,"frontend_base_url":"http://10.134.32.132:5273","backend_base_url":"http://10.134.32.132:8266"}]
IDS_GATEWAY_SITE_MAP=[]
```

如果第二个网站是单体服务，例如 `8080`，那么这里改成：

```env
IDS_GATEWAY_PORT_MAP=[{"site_key":"site2","display_name":"site2","ingress_port":8288,"frontend_base_url":"http://10.134.32.132:8080","backend_base_url":"http://10.134.32.132:8080"}]
```

## 7. 配置完成后的访问方式

配置完成以后，客户端统一访问 IDS 机器：

- 网站 1：`http://10.134.32.142:8188`
- 网站 2：`http://10.134.32.142:8288`
- IDS 控制台：`http://10.134.32.142:5175`

不要再让客户端直接访问网站机器的业务端口，例如：

- `http://10.134.32.132:5173`
- `http://10.134.32.132:8166`
- `http://10.134.32.132:5273`
- `http://10.134.32.132:8266`

## 8. 防绕过要求

如果网站机器的业务端口仍然对所有来源开放，那么别人还是可以绕过 IDS，直接访问网站机器。

所以真实部署时要满足这条原则：

- 网站机器业务端口只允许来自 IDS 机器 IP 的访问

也就是只允许：

- `10.134.32.142 -> 10.134.32.132:5173`
- `10.134.32.142 -> 10.134.32.132:8166`
- `10.134.32.142 -> 10.134.32.132:5273`
- `10.134.32.142 -> 10.134.32.132:8266`

如果第二个网站是 `8080`，就把上面的 `5273 / 8266` 改成 `8080`。

## 9. 验证方式

先访问正常页面：

- `http://10.134.32.142:8188`
- `http://10.134.32.142:8288`

再测试恶意请求是否被网关拦截：

```text
http://10.134.32.142:8188/?id=1%20union%20select%201
```

或者：

```text
http://10.134.32.142:8288/?id=1%20union%20select%201
```

预期结果：

- 浏览器收到 `403`
- IDS 会话记录里出现对应来源 IP
- IDS 事件中心出现告警
- 如果手工封禁某个来源 IP，该 IP 访问两个入口都会被统一拒绝

## 10. 这套方式已经实现了什么

当前已经实现的是：

- 一个 IDS 机器可以接多个网站入口
- 每个入口端口可独立转发到不同网站
- 所有请求都会先进入 IDS 再决定放行或拦截
- 规则命中后会直接返回 `403`
- IP 被封禁后，再访问 IDS 监管下的入口会被统一拒绝
- 访问与拦截记录都会写入 IDS 会话和事件数据

## 11. 什么时候不适合这套方案

下面这种需求不适合当前这套端口映射方案：

- 想在没有域名和 `hosts` 的前提下，让同一个 IP 的同一个端口承载多个网站
- 想让 IDS 看到外部互联网任意站点的所有流量

如果以后要做到“同一个 IP、同一个 80/443 端口承载多个站点”，就必须引入：

- 域名
- 反向代理
- 或者更底层的透明代理/旁路镜像方案
