# 旧站参考入口清单

## 1. 用途说明

这份清单只服务于一件事：

从“添加前的网站”里提炼出值得参考的接入点样本，帮助未来新版网站对接独立 IDS。

它不代表未来新版网站一定还长这样，也不应该被当成最终接口契约。

## 2. 旧站里最值得保留参考价值的入口

### 2.1 登录入口

前端公开路由：

- `/login`

后端接口：

- `POST /api/user/login`
- `GET /api/user/info`

补充说明：

- 旧站账号管理也仍挂在 `/api/user/*` 下
- 兼容过 `system_admin` / `admin` 这类历史角色映射

参考价值：

- 可以拿来理解旧站的登录链路、角色粒度和令牌使用方式
- 但不应该直接作为未来新版网站的最终身份方案

### 2.2 公共上传入口

前端公开路由：

- `/upload`

前端页面：

- 匿名上传页 `PublicUpload.vue`

后端接口：

- `POST /api/upload`
- `GET /api/upload/quarantine`
- `POST /api/upload/quarantine/analyze`
- `GET /api/upload/quarantine/{filename}/report`
- `DELETE /api/upload/quarantine/{filename}`

参考价值：

- 这是旧站里最像“全站统一文件扫描入口”的一条链路
- 非常适合作为未来新版网站统一上传扫描能力的样本

### 2.3 审计入口

前端路由：

- `/audit`

后端接口：

- `GET /api/audit`

权限特点：

- 仅管理员可查看

参考价值：

- 可以作为未来“业务审计回传 -> IDS 汇总展示”的样本
- 但不建议未来继续把安全审计和业务审计硬绑在同一个站内页面

### 2.4 用户管理入口

前端路由：

- `/user`

后端接口：

- `GET /api/user/manage`
- `POST /api/user/manage`

权限特点：

- 仅管理员可管理

参考价值：

- 可以帮助我们理解旧站角色和用户管理方式
- 但未来更建议 IDS 用户体系独立，或者等新版网站稳定后再做 SSO

### 2.5 安全中心入口

前端路由：

- `/security/ids`
- `/security/log-audit`
- `/security/situation`
- `/security/sandbox`

旧站内还保留了兼容跳转：

- `/ids` -> `/security/ids`

参考价值：

- 可以帮助我们提炼旧站中 IDS 的页面构成
- 但未来不应再要求 IDS 必须嵌在业务站内部路由下

## 3. 旧站里应重点关注的“敏感业务面”

虽然未来新版网站可能变化很大，但下面这些区域通常仍然值得重点关注，因为它们天然更容易和安全事件、权限变更、附件流转、后台操作关联。

### 3.1 后台管理面

参考路由：

- `/user`
- `/audit`
- `/security/*`
- `/supplier`
- `/purchase`
- `/warning`

原因：

- 涉及权限、配置、管理动作、告警处置
- 更适合接入操作审计和敏感行为告警

### 3.2 文件与单据流转面

参考路由：

- `/upload`
- `/purchase`
- `/stock/in`
- `/stock/out`
- `/delivery`
- `/supplier/orders`

原因：

- 往往伴随附件上传、文档导入、报表导出、图片/PDF/Excel 流转
- 未来新版网站如果加了更多内容，这类区域大概率只会更多，不会更少

### 3.3 公开访问面

参考路由：

- `/login`
- `/upload`
- 其他任何无需登录即可访问的页面

原因：

- 这是最容易直接暴露给外部请求探测和攻击流量的区域
- 未来网关/WAF 接入优先覆盖这里

## 4. 从旧站可以提炼出的未来接入提示

### 4.1 登录不要深耦合旧链路

旧站登录链路可以提供参考，但未来新版网站应重新确认：

- 是否仍然是站内登录
- 是否引入统一身份认证
- 是否保留相同角色体系

所以现在最多只保留“角色大类”和“令牌鉴权思路”的参考，不直接复用旧站用户表。

### 4.2 上传链路最值得优先保留

旧站的 `/upload -> /api/upload` 这条链路，是最适合延续到未来新版网站的一段样本。

未来建议直接固化成规则：

- 所有上传能力统一接入独立 IDS 扫描
- 无论新版网站新增多少上传页面，都不要再各写一套检测逻辑

### 4.3 安全中心页面不要再嵌回业务站

旧站里 IDS 被挂在：

- `/security/ids`
- `/security/log-audit`
- `/security/situation`
- `/security/sandbox`

未来建议：

- 这些页面继续存在
- 但放在独立 IDS 前端里
- 业务站只负责跳转或通过链接进入，不再负责承载 IDS UI

### 4.4 审计要从“站内查看”转成“独立汇聚”

旧站有：

- `/api/audit`
- `/audit`

未来更合适的方式是：

- 业务站继续产生日志
- 关键安全审计回传给独立 IDS
- IDS 做统一汇聚、检索、导出和告警

## 5. 旧站参考入口 -> 未来接入边界映射

### 5.1 登录样本

旧站样本：

- `/login`
- `/api/user/login`

未来边界：

- 作为“身份方案待确认项”
- 暂不作为固定契约

### 5.2 上传样本

旧站样本：

- `/upload`
- `/api/upload`

未来边界：

- 直接升级为“全站统一文件扫描契约”

### 5.3 审计样本

旧站样本：

- `/audit`
- `/api/audit`

未来边界：

- 升级为“业务审计回传 -> IDS 汇聚”

### 5.4 安全中心样本

旧站样本：

- `/security/ids`
- `/security/log-audit`
- `/security/situation`
- `/security/sandbox`

未来边界：

- 页面能力保留
- 部署位置独立

## 6. 现在可以据此直接做的事

1. 把“统一上传扫描”定为未来新版网站的硬规则。
2. 把“安全事件统一投递 IDS”定为未来新版网站的硬规则。
3. 把“业务审计回传到 IDS”定为未来新版网站的推荐规则。
4. 把“安全中心独立部署”定为架构原则，不再嵌回业务站。

## 7. 一句话结论

旧站最值得留下来的，不是它的页面结构，而是它暴露出来的几类入口样本：

- 登录入口
- 上传入口
- 审计入口
- 安全中心入口

未来新版网站真正应该延续的，是这些入口背后的“接入边界”，而不是旧站当时那套具体实现。
